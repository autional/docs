# 文档站品牌化改造详细变更说明

> 记录日期：2026-06-30  
> 项目：`autional-docs-site`  
> 变更类型：品牌视觉升级、共享布局重构、首页门户化、正文样式统一、品牌资源接入  
> 适用范围：文档站共享布局、首页、正文样式体系、SDK 页面代码块修复、品牌资源

---

## 1. 目标是什么

本次变更的目标，是让 `autional-docs-site` 从“可用的 Astro 文档站”升级为“和主官网品牌一致、同时适合长期扩展的产品文档门户”。

核心目标包括：

1. 让文档站与主官网 `autional-web` 的品牌语言统一。
2. 提升首页与共享布局的完成度，让文档站不再显得过于简陋。
3. 通过全局正文样式层，统一所有文档页的标题、段落、表格和代码块样式。
4. 接入统一品牌 Logo 与 favicon。
5. 修复 SDK 页面中代码块示例的渲染问题。

---

## 2. 变更前的问题

### 2.1 共享布局过于基础

文档站原布局只有：

- 左侧侧栏
- 右侧正文
- 极简标题与少量链接

存在的问题包括：

- Header 缺少品牌感
- 页面整体像默认文档站壳子
- 与主官网缺乏统一识别

### 2.2 没有全局正文样式体系

项目中原本没有 `global.css` 这一类统一样式入口，导致：

- 表格样式分散
- 代码块样式分散
- 正文标题层次缺少统一标准
- 新页面难以保证一致性

### 2.3 首页只是普通 Quick Start 内容页

首页内容虽然正确，但视觉与结构都更像单篇文档，而不是“文档门户首页”。

问题包括：

- 缺少入口聚合能力
- 缺少显式引导
- 缺少门户感

### 2.4 SDK 页面代码块渲染错误

在 `/sdk` 页面中，React 示例代码使用了 `set:html` 渲染包含 JSX 标签的字符串，导致：

- `<AuthProvider>` 等内容被浏览器当成真实标签解析
- `return (...)` 内部文字不可见

这是一个影响可读性的实际问题，不只是视觉瑕疵。

## 3. 本次实际做了什么

### 3.1 新增全局品牌样式文件

新增：

- `src/styles/global.css`

该文件承载了文档站新的样式基础设施，包括：

- `brand-shell`
- `brand-card`
- `brand-button-primary`
- `brand-button-secondary`
- `brand-kicker`
- `docs-nav-link`
- `docs-prose`

其中 `docs-prose` 负责统一：

- `h1 / h2 / h3`
- `p`
- `ul / ol`
- `a`
- `pre`
- `code`
- `table`
- `blockquote`

这样做的原因是：

- 文档站更适合通过“正文规范层”来统一全站
- 比逐页手工修样式更稳定、更可维护

### 3.2 扩展 Tailwind 品牌令牌

更新：

- `tailwind.config.cjs`

补充与主官网一致的：

- `primary`
- `sky`
- `amber`
- 字体栈
- 阴影体系

这样处理的目的是，避免文档站继续使用默认蓝色系，而是直接接入 Autional 自己的品牌主题。

### 3.3 重构文档站共享布局

更新：

- `src/layouts/DocLayout.astro`

具体改动：

- 引入全局样式
- 接入品牌字体
- 新增顶部 Header
- 重做侧栏容器
- 统一正文外层容器
- 接入品牌 Logo 与 favicon

重构后的布局逻辑是：

- Header 负责品牌识别
- 左栏负责文档导航
- 正文负责统一内容展示

### 3.4 首页门户化

更新：

- `src/pages/index.astro`

首页不再只是一个普通 Quick Start 页面，而是升级为：

- 顶部文档门户引导
- 三步启动卡片
- 常用入口卡片集合

采用这种方式的原因：

- 文档首页不仅要承载第一篇文档，还应该承担“导航枢纽”的角色
- 用户进入文档站时，应该快速知道从哪里开始，以及下一步去哪里

### 3.5 修复 SDK 页面代码块问题

更新：

- `src/pages/sdk.astro`

问题原因：

- 使用 `set:html={reactCode}` 渲染 JSX 字符串

修复方式：

- 改为 `<pre><code>{reactCode}</code></pre>` 这种普通文本代码渲染

为什么这样改：

- 可以避免浏览器把 JSX / HTML 风格内容当成真实节点解析
- 保证 `function App() { return (...) }` 等示例完整可见

### 3.6 接入统一品牌资源

新增：

- `public/logo-mark.svg`
- `public/favicon.svg`

这样做的原因，是让文档站与主官网、开发者站共享同一套品牌图形资产。

## 4. 为什么采用这些方式

### 4.1 为什么重点改布局和正文样式

文档站不是营销站。  
对于文档站来说，最重要的不是把每一页都做成营销大首页，而是：

- 布局统一
- 正文可读
- 导航清晰
- 首页有门户感

因此本次优先策略是：

1. 建立正文样式体系
2. 重构共享布局
3. 门户化首页
4. 修复实际渲染问题

### 4.2 为什么不是逐页重写所有文档

文档站页面较多，如果逐页重写，成本高且收益不成比例。

通过统一 `DocLayout + docs-prose`，可以一次性让所有页面一起变好看，这是更高性价比的方案。

### 4.3 为什么 SDK 页面不用 `set:html`

只要内容里有 JSX 风格标签，`set:html` 就存在把代码解析为节点的风险。  
对于代码示例，普通文本渲染更安全、更稳定。

## 5. 遇到的困难与解决方式

### 5.1 文档站原本没有全局正文规范

问题：

- 样式分散
- 维护困难
- 新页面容易不一致

解决：

- 新增 `src/styles/global.css`
- 把通用正文样式集中到 `docs-prose`

### 5.2 SDK 代码块不是单纯配色问题

用户反馈是 `return` 内部“看不见”，表面像颜色问题。  
实际检查后发现，根因是 JSX 标签被当成真实 DOM 解析。

解决：

- 不从颜色入手，而是先修正代码渲染方式

这次问题也说明，视觉问题有时实际上是渲染逻辑问题。

### 5.3 本地实时预览受环境限制

与开发者站类似，当前运行环境对 `.astro/types.d.ts` 写入有限制，导致 `astro dev` 无法在本环境中顺畅打开。

解决：

- 以 `npx astro check` 作为主要验证手段
- 将其记录为环境限制，不误判为项目代码错误

## 8. 新发现的技术债

### TD-1：移动端文档导航仍缺少更强的交互形态

当前侧栏在小屏环境下虽然仍能显示内容，但还没有真正意义上的折叠导航、抽屉或目录切换方案。

影响：

- 在移动端阅读长文档时，导航体验仍有提升空间

建议：

- 后续单独设计移动端文档导航方案

### TD-2：文档代码块目前仍是统一深色面板，未引入正式语法高亮

当前代码块已经可读，但未接入 token 级语法高亮系统。

影响：

- 可用性达标，但精致度和专业感还有进一步提升空间

建议：

- 后续可结合语法高亮方案继续升级

## 9. 经验教训

### Lesson-1：文档站最重要的是统一正文规范

对文档站来说，统一正文样式比零散页面美化更重要。  
只要正文层规范做好，大部分页面都会一起受益。

### Lesson-2：视觉问题不一定是颜色问题

SDK 页这次暴露的问题很好地说明：

- 用户看到的是“看不见”
- 根因却可能是“渲染方式错误”

所以遇到这类问题时，必须先确认渲染链路是否正确。

### Lesson-3：布局层是文档站升级的最高杠杆点

相比逐页修饰，优先升级共享布局和正文容器，是更高收益的做法。
